Criminal Site Selection (CSS) Modeling in R

Sam Huddleston April 29, 2016

This lab provides an introduction to the development of Criminal Site Selection (CSS) models using R statistical software. CSS models are a technique used to develop hot-spot maps to predict future criminal activity. CSS models have been shown to significantly improve predictive performance over traditional kernel density hot-spot methods for predicting crimes such as burglaries [1,2], terrorist events [3], suicide bombings [4], and criminal activity by street gangs [5,6]. It has also been shown to be useful for geographic profiling - finding the anchor point (home or workplace) of criminal offenders [7].

This tutorial consists of six basic steps:

  1. Set up the R Programming Environment
  2. Import and Organize the Raw Data Files
  3. Develop CSS Modeling Data Frames
  4. Develop Criminal Site Selection Models
  5. Evaluate CSS Model Performance
  6. Develop Maps of CSS Model Predictions

Step 2: Import and Organize the Raw Data Files

In this lab, we’ll be using the R programming language as a Geographic Information System (GIS). The next step in this process is to bring in the geographic shapefiles that contain the raw data we will use to build and visualize the CSS models. In the shapefiles folder, there are seven shapefiles (note that the gang names have been anonymized in this dataset):

  • SA_Outline: an outline of the City of Santa Ana, California

  • Gang_Points: a gang intelligence map containing the point locations (addresses) for some of the criminal gangs in the city.

  • Gang_Territory: a gang intelligence map containing the “claimed” gang territories for some of the criminal gangs in the city.

  • Block_Groups: demographic information at the Census block group level from the 2000 Census

  • Streets: a line shapefile street map of the City of Santa Ana, California

  • Incidents: a point shapefile of the locations of criminal incidents by the six most violent gangs in the city

  • Grid_Points: a point shapefile of a point grid at a high resolution over the City of Santa Ana, which will be used for modeling and visualization

Step 2b: Plotting Shapefiles

Once you have read in a shapefile, you can use the basic plot() function to create a map. You can layer shapefiles using the “add = TRUE” as below:

plot(SA.Outline)
plot(Gang.Territory, add=TRUE, col="blue")
plot(Incidents, add=TRUE, col="red", pch=20)

Notice in the map above that one of the three layers we attempted to plot does not seem to appear. There doesn’t seem to be any gang territories plotted. There seems to be a problem (and there is). Our map layers are not lining up properly.

Step 2c: Transform All Shapefiles to the Same Coordinate System / Projection

The issue here is that geographic information stored in GIS can be stored in many different model formats. Two different shapefiles may be using different models of the world and/or they may be stored in different projections. In order to learn what model the shapefile has, we can use the proj4string() function from the sp package.

proj4string(SA.Outline)
## [1] "+proj=lcc +lat_1=32.78333333333333 +lat_2=33.88333333333333 +lat_0=32.16666666666666 +lon_0=-116.25 +x_0=2000000 +y_0=500000.0000000002 +datum=NAD83 +units=us-ft +no_defs +ellps=GRS80 +towgs84=0,0,0"
proj4string(Gang.Territory)
## [1] "+proj=longlat +datum=NAD83 +no_defs +ellps=GRS80 +towgs84=0,0,0"

Note that these two models/projections are different. We need to transform the Gang.Territories map layer to the same coordinate system as our SA.Outline layer. The spTransform() function provides a way to do this. The code block below transforms all of our map layers to the same projection and coordinate system.

Gang.Points<-spTransform(Gang.Points, CRS(proj4string(SA.Outline)))
Block.Groups<-spTransform(Block.Groups, CRS(proj4string(SA.Outline)))
Streets<-spTransform(Streets, CRS(proj4string(SA.Outline)))
Gang.Territory<-spTransform(Gang.Territory, CRS(proj4string(SA.Outline)))
Incidents<-spTransform(Incidents, CRS(proj4string(SA.Outline)))

Now, when we plot our map, all of the layers appear. Note that the Block.Groups layer will cover all other layers when plotted because the grid used is very fine (there are more than 40,000 points).

plot(SA.Outline)
plot(Gang.Territory, add=TRUE, col="blue")
plot(Streets, add=TRUE)
plot(Gang.Points, add=TRUE, col="green", pch=15)
plot(Incidents, add=TRUE, col="red", pch=20)
plot(Block.Groups, add=TRUE)

Step 3: Develop CSS Modeling Data Frames

For this lab, we are building CSS models that leverage logistic regression and random forest classification models. The models models require a training dataset that contains observations of where criminals (or criminal gangs) chose to commit crimes and observations of where criminals chose not to commit crimes. Therefore, we will have to create two spatial datasets. The first is a dataframe associated with criminal incidents containing spatial predictor variables and a response variable (Response = 1) denoting the occurrence of crime. The second datafame is the “Null Grid” dataframe that represents non-occurrence (Response = 0). We’ll begin by creating the crime occurrence dataset and adding feature space, categorical, and demographic predictors from our spatial datasets.

Step 3a: Create a Response Dataframe

The code block below creates a new spatial layer titled “Inc.Data” that has only one feature in its’ dataframe. Note that we can use the “@” operator to assign elements to this spatial layer’s dataframe. Geographically this layer is identical to our Incidents data layer, but it has only one column in its’ dataframe.

Response<-data.frame(rep(1, length(Incidents@data[,1])))
colnames(Response)<-"Response"
Inc.Data<-Incidents
Inc.Data@data<-Response

Step 3b: Add Feature Space Predictors

The next step is to begin adding predictor variables to the Inc.Data dataframe. The first two predictor variables we will add are feature space predictors. Feature space predictors measure the distance between an observation and key geographic features. For this lab, we’ll measure the distance to the Gang.Points spatial point layer and the Streets spatial line layer. The nncross() function from the spatstat package provides the ability to measure the distance to the nearest point or line feature. Note the use of the as.ppp() and as.psp() functions to denote the difference between the point (as.ppp) and line (as.psp) layers.

DistToGangPnts<-nncross(as.ppp(Inc.Data), as.ppp(Gang.Points))[,1]
DistToStreet<-nncross(as.ppp(Inc.Data), as.psp(Streets))[,1]

We now have two new predictor variables for our Inc.Data dataframe stored as vectors. We’ll add them to the dataframe later. For right now, let’s take a look at one of the predictors using the hist() function.

hist(DistToGangPnts)

Taking a look at this histogram for DistToGangPnts, we see the motivation for using logistic regression for modeling this problem: the incidence of gang crimes decreases with a classic logistic curve as we move away from gang headquarter locations (as one would expect). Take a second to plot the effect of DistToStreet - this effect is even stronger.

Step 3c: Add Categorical Predictors

The next predictor variable we’ll consider is a categorical predictor: Does this incident fall within (or without) a known gang territory? This can be easily checked using the over() function from the sp package. The over function returns an “NA” value whenever the point layer doesn’t fall within any one of the polygon layers we are matching against. We can exploit that fact by using the !is.na() function (which literally says, “find things NOT NA”) to create a categorical 1 or 0 predictor variable indicating whether or not an incident falls within a gang territory.

Territory.Check<-over(Inc.Data, Gang.Territory)[,1]
InGangTerr<-vector(mode="numeric", length=length(Territory.Check))
InGangTerr[!is.na(Territory.Check)]<-1

Step 5: Evaluate CSS Model Performance

A surveillance plot records the amount of surface area predicted by a model a security agency would need to monitor in order to observe a corresponding percentage of incidents occurred and is an adaptation of a model comparison metric introduced by Smith and Brown [4]. A modelâs tradeoff in accuracy and precision as shown in a surveillance plot is a very important characteristic for evaluating geospatial model performance because we desire a model that focuses security efforts in as small a geographic area as possible.

A surveillance plot is created by rank ordering all of the points on a modelâs general predicted threat surface in descending order from the most likely to the least likely location. The horizontal axis of the surveillance plot records the percentage of surface area, referenced in descending order of threat surface, needed to observe the corresponding percentage of actual incidents in the data set.

The surveillance plot evaluates the performance of the model in the same way that an ROC curve does, except that it couches the evaluation in terms of evaluating the efficiency of a model in allocating resources expended by a security agency in attempting to observe or interdict the crimes as they occur. The best model is the one that achieves the highest percentage of observed incidents for a given percentage of high probability surface area required. The appropriate point of evaluation on the surveillance plot graph is contingent upon the resources available to the evaluating agency. For instance, if it is possible to monitor only 5% of the surface area in the research area with your available resources, then the model that provides the highest number of observed incidents in the test set in 5% or less of the high-probability surface area observed is the best one for your organization.

The code block below creates a function that can be used to generate a surveillance plot for a given model.

survplot<-function(prediction, response){
  SurvPlotData<-cbind(prediction,response)
  SurvPlotData2<-SurvPlotData[order(-prediction),]
  SurvPlotPerc<-vector(mode="numeric", length=length(prediction))
  SurvPlotArea<-vector(mode="numeric", length=length(prediction))
  total<-sum(response)
  for (i in 1:length(prediction)){
    SurvPlotPerc[i]<-sum(SurvPlotData2[1:i,2])/total
    SurvPlotArea[i]<-i/length(response)
  }
  SurvPlot<-cbind(SurvPlotArea, SurvPlotPerc)
  SurvPlot}

This plot is not that useful as it is hard to see exactly what is going on. However, we can use the leaflet package to build an interactive visualization of our prediction. The RColorBrewer package allows us to quickly build a color palette for our predictions using the brewer.pal() function. We can use the htmlwidgets package (called by leaflet) to write out our map to an html page that can be shared with others. In the example below, we use a nice greyscale street map tile called Stamen.Toner to create the map background.

library(leaflet)
library(RColorBrewer)

#Set the projection for our raster and street map                  
crs(Prediction.Raster) <- sp::CRS(proj4string(SA.Outline))    

#Add color palette using RColorBrewer
palette<-brewer.pal(8, "Reds")
pal <- colorNumeric(palette, values(Prediction.Raster),
                    na.color = "transparent")

#Define Lat-Long for our plot using extent()
Latitude=mean(extent(Streets)[1:2])
Longitude=mean(extent(Streets)[3:4])
                   
#Plot using leaflet()                 
my.map<-leaflet() %>%
   setView(Latitude, Longitude, zoom = 12) %>%
   addProviderTiles("Stamen.Toner") %>%
   addRasterImage(Prediction.Raster, colors = pal, opacity = .8) %>%
   addLegend(pal = pal, values = values(Prediction.Raster),
         title = "Prediction") 
my.map   #to plot the map

#To save the map as an html
htmlwidgets::saveWidget(mymap, "SALeafletGangMap.html", selfcontained=TRUE)
