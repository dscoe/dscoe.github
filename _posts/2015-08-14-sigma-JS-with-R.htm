Introduction

This demo will show how to use the Sigma JS graphing library with R. As the name implies, Sigma JS is a javascript library that draws graphics. Sigma scales much better than D3 and is much faster when rendering graphs. Also, Sigma includes many options for interactivity including zoom and on-click events. Check out the Sigma website for demos and additional information.

While Sigma is a great resource, there is no straight forward way to integrate it with R. There is a community working to connect the two, but currently there is only minimal support and almost no documentation. This demo will walk through the data processing and rendering of basic Sigma graphics, but keep in mind that Sigma/R integration will likely continue to be developed.

The basis for Sigma/R integration is from an example on the HTML widgets website. Briefly, HTML Widgets is a package that supports a connection between R and javascript libraries. If you are familiar with rCharts, the basic idea here is the same.

Because Sigma/R capabilities are still being developed, you will need to install the R sigma package using devtools:

install.packages(devtools::install_github("jjallaire/sigma"))

You can now load the sigma package. Since there is no documentation included in the package, you may want to go to the authors github page to check out the examples.

The main function within the sigma package is sigma(). This function takes a gexf file (more explanation below) and renders a sigma graph. It is important to note that Sigma JS is a graph drawing library, not a graphing library. In other words, Sigma JS is great a drawing graphs, but is not able to learn graphs. As a result, you will need to specify all of the characteristics of the graph you want to render to Sigma JS first.

The rest of this demo will follow the major steps necessary to draw Sigma graphs in R: 1. Learn graph specification from data 2. Convert graph specification to GEXF format 3. Draw graph using Sigma

Graphing with Sigma

Step 0: Data Preparation

The most important step in this process is data preparation. If you run into a problem down the line, the first place to look for troubleshooting is the data preparation.

The desired end result is two dataframes: edges and nodes. The node dataframe must have 2 columns: id and label where id is a unique number and label is a text description. The edges dataframe must also have 2 columns: source and target which are both numeric and correspond to specific node ids. Each pair of source and target represents an edge between the referenced nodes.

For this demo, I will use data from the most recent world cup (soccer). This data set has information on the player level. The graph that I will make from this data will relate world cup teams to clubs. An edge will represent a player from a country who is connected to a given club. The basic insight from the graph will be which countries tend to send players to the same clubs and/or where top club players play their international soccer.

First we will read the data and narrow down to the two variables we are interested in:

Step 1: Learn Graph

There are may ways to learn graphs in R, but this demo will use the sna package. The gplot.layout function works with different algorithms to specify specific node positions. For this example, we will use the Fruchterman-Reingold algorithm.

First, we need to convert the data to a matrix with rows and columns for each node. The matrix will be equal to 1 if there is a connection between the associated nodes.

links <- matrix(rep(0, nrow(nodes)*nrow(nodes)), ncol = nrow(nodes))
for(i in 1:nrow(edges)){
  links[(edges[i,]$source), (edges[i,]$target)] <- 1
}

positions <- gplot.layout.fruchtermanreingold(links, layout.par=list())
positions <- cbind(positions, 0) # need a z axis for the write.gexf() formula later

Step 2: Convert Data to GEXF

GEXF format is basically a list of dataframes and vectors that explain a graphs attributes. The rgexf package is very particular about the inputs to the conversion formula: write.gexf. As a result, we will need a few additional data elements before converting what we already did.

To recap, we already have an edge list, a node list, and the positions of our nodes (learned using the Fruchterman-Reingold algorithm). Next we need to make the node color, edge color, and edge size attributes.

All of the node attributes need to be the same length as the node list (the same 1:1 relationship also must exist for the edge attributes). To make the node and edge colors, we have to stipulate the red, green, blue, and alpha (opacity) variables, then create a dataframe with a row corresponding to each node/edge.

color.red <- rep.int(150, nrow(nodes))
color.green <- rep.int(1, nrow(nodes))
color.blue <- rep.int(1, nrow(nodes))
color.alpha <- rep.int(1, nrow(nodes))
color.node <- data.frame('red'=color.red, 'green'=color.green, 'blue'=color.blue, 'alpha'=color.alpha)

color.node$green[nodes$label %in% unique(df.sub$Team)] <- 150
color.node$red[nodes$label %in% unique(df.sub$Team)] <- 1

color.edge.red <- rep.int(1, nrow(edges))
color.edge.green <- rep.int(1, nrow(edges))
color.edge.blue <- rep.int(50, nrow(edges))
color.edge.alpha <- rep.int(.2, nrow(edges))
color.edge <- data.frame('red'=color.edge.red, 'green'=color.edge.green, 'blue'=color.edge.blue, 'alpha'=color.edge.alpha)

Similarly, we need to specify the size of the nodes. Here we will use the number of edges associated with a node to determine its size. We can then scale the frequencies from 0 to 1.

size.count <- count(node.list)
colnames(size.count) <- c('node','freq')
size.merge <- merge(x=nodes, y=size.count, by.x='label',by.y='node', all.x=TRUE)
size.sort <- size.merge[order(size.merge$id),]
size <- size.sort[,3]
size <- (size-min(size))/(max(size)-min(size))

We now have all of the building blocks of a GEXF file and we can now combine them all into a GEXF file using the rgexf package. Note, the below code will output the graph to the working directory (the preferred method).

graph <- write.gexf(nodes=nodes,
                    edges=edges,
                    nodesVizAtt=list(
                      position=positions,
                      color=color.node,
                      size=size),
                    edgesVizAtt=list(
                      color=color.edge),
                    output='soccerGraph.gexf')

Step 3: Draw Graph

Now that we have a graph in a GEXF file, we can simply run the sigma() function to render the graph.

Because sigma is an underdeveloped package in R, you will not get any errors if the graph fails to render - leaving you with no way to know where you may have gone wrong. The key is to make sure that you have all of the attributes that sigma() needs to render the graph: nodes, edges, a position for each node, a color for each node, and a size for each node. Edge attributes are optional.

sigma('soccerGraph.gexf', width=900, height=1000)

Note that you can zoom, click and explore the graph.

In this graph, the green nodes represent national teams and the red nodes represent club teams. You can see that the bigger soccer countries are more central (forgive that the names are in German), as are the most popular soccer clubs. This makes sense: England, Germany, and Italy send many players to top clubs such as Bayern Munich, Manchester United, and Chelsea. Looking at the least central national teams, we can see that countries like Iran share fewer players with top clubs.

Sigma can also be easily integrated into shiny apps using renderSigma() in the server and sigmaOutput() in the UI.

gexf <- system.file("examples/ediaspora.gexf.xml", package = "sigma")

ui = shinyUI(fluidPage(
  sigmaOutput('sigma', height=1000)
))

server = function(input, output) {
  output$sigma <- renderSigma(
    sigma('soccerGraph.gexf', width=1000, height=1000)
  )
}

shinyApp(ui = ui, server = server)

Looking Forward

Integration between R and GEXF/Sigma is limited, but growing. The rgexf package is under active development, which could soon lead to the ability to specify many more attributes.

Unfortunately, the sigma package is not under development. It was provided as an example of HTML widget use, so it is not actually intended to be serviced as a normal package. As a result, it may be necessary to develop add-ons ourselves to support additional Sigma functionality. Sigma JS supports many additional interactions and functional improvements such as on-click events and dynamic graph subsetting. It will take some time working with HTML widgets, R, and Sigma JS to further this development.
